# SCOOP → Loop 통합 로드맵

**작성일:** 2026-02-13
**핵심 원칙:** SCOOP이 Loop의 기반이 된다. 별도 앱이 아닌 통합 운영.

---

## 전략 요약

SCOOP 앱은 Loop 플랫폼의 **Phase 0(기반 시스템)**으로서, 이미 구현된 주행 기록, 그룹 라이딩, 소셜 기능 위에 마켓플레이스, 정비 관리, 결제 시스템을 점진적으로 확장한다. 완전히 새로 만드는 것이 아니라, 기존 SCOOP 코드베이스를 Loop로 진화시키는 방식이다.

---

## 현재 SCOOP에서 이미 구현된 것 (Loop 관점)

SCOOP에 이미 존재하는 기능들은 Loop의 핵심 기능과 직접 대응된다. 아래 표는 SCOOP의 기존 기능이 Loop의 어떤 영역을 커버하는지 정리한 것이다.

| Loop 기능 영역 | SCOOP 현재 상태 | 비고 |
|---------------|----------------|------|
| 사용자 인증 (OAuth) | 구현 완료 | Google, Kakao, Apple 소셜 로그인 |
| 기기 등록/관리 | 구현 완료 | 브랜드, 모델, 닉네임 등록 |
| 주행 기록 (GPS 추적) | 구현 완료 | 자동 거리/속도/시간 기록 |
| 그룹 라이딩 | 구현 완료 | 생성, 참가, 실시간 위치 |
| 친구 시스템 | 구현 완료 | 친구 추가, 검색, 프로필 |
| 배지/업적 시스템 | 구현 완료 | 주행 기반 배지 |
| AI 주행 분석 | 구현 완료 | Gemini API 기반 |
| 관리자 대시보드 | 구현 완료 | 사용자/기기/주행 관리 |
| 푸시 알림 | 구현 완료 | Expo Notifications |
| HUD (속도계) | 구현 완료 | 주행 중 실시간 표시 |

## Loop로 확장하기 위해 필요한 것

### Phase 1: 마켓플레이스 기반 (예상 2-3개월)

마켓플레이스는 Loop의 핵심 수익 모델이다. 부품 호환성 DB를 구축하고, 상품 검색/구매/결제 흐름을 구현해야 한다.

| 작업 | 설명 | 난이도 |
|------|------|--------|
| 호환성 DB 설계 | device_catalog, compatibility_groups, product_compatibility 테이블 | 높음 |
| 상품 CRUD API | 상품 등록, 검색, 상세, 호환성 필터링 | 중간 |
| 장바구니 시스템 | 장바구니 담기, 수량 변경, 삭제 | 중간 |
| Stripe 결제 연동 | 주문 생성, 결제 확인, Webhook | 높음 |
| 주문 관리 | 주문 내역, 배송 추적, 수령 확인 | 중간 |
| 리뷰 시스템 | 상품 리뷰, 별점, 이미지 | 낮음 |

**기술 변경 필요:**
- DB: MySQL → PostgreSQL 마이그레이션 (또는 MySQL 유지 후 나중에 전환)
- 인프라: GCP → AWS Seoul (또는 GCP 유지)
- 결제: Stripe SDK 추가

---

### Phase 2: 정비 관리 시스템 (예상 1-2개월)

SCOOP의 주행 기록 데이터를 활용하여 자동 정비 알림을 구현한다. 주행 거리 기반으로 소모품 교체 시기를 자동 계산하고, 부품 구매로 연결하는 것이 핵심이다.

| 작업 | 설명 | 난이도 |
|------|------|--------|
| 정비 유형 DB | maintenance_types (타이어, 브레이크, 배터리 등) | 낮음 |
| 자동 알림 생성 | 주행 거리 기반 정비 알림 크론잡 | 중간 |
| 정비 이력 기록 | 정비 완료 기록, 다음 알림 자동 생성 | 중간 |
| 정비소 검색 | 지도 기반 근처 정비소 표시 | 중간 |
| 알림 → 구매 연결 | 정비 알림에서 호환 부품 바로 구매 | 낮음 |

---

### Phase 3: 실시간 기능 강화 (예상 1-2개월)

SCOOP의 그룹 라이딩 기능을 WebSocket 기반 실시간 채팅과 위치 공유로 업그레이드한다.

| 작업 | 설명 | 난이도 |
|------|------|--------|
| WebSocket 서버 | Redis Pub/Sub + WebSocket 서버 구축 | 높음 |
| 그룹 채팅 | 실시간 메시지, 입장/퇴장 알림 | 중간 |
| 1:1 채팅 | 친구 간 실시간 메시지 | 중간 |
| 실시간 위치 브로드캐스트 | 그룹 라이딩 중 30초마다 위치 공유 | 중간 |

---

### Phase 4: 판매자 시스템 (예상 2-3개월)

마켓플레이스의 공급 측면을 구축한다. 판매자 온보딩, 상품 관리, 주문 처리, 정산 시스템이 필요하다.

| 작업 | 설명 | 난이도 |
|------|------|--------|
| 판매자 가입/인증 | 사업자 등록증 확인, 계좌 등록 | 중간 |
| 판매자 대시보드 | 주문 관리, 매출 통계, 상품 관리 | 높음 |
| 정산 시스템 | Stripe Connect, 수수료 계산, 정산 주기 | 높음 |
| 프로모션 관리 | 할인 코드, 프로모션 설정 | 중간 |

---

### Phase 5: 글로벌 확장 (예상 3-6개월)

서울에서 검증된 후, 샌프란시스코 → 뉴욕 → 유럽 순으로 확장한다.

| 작업 | 설명 | 난이도 |
|------|------|--------|
| 다국어 지원 | i18n (한국어, 영어) | 중간 |
| 해외 결제 | 통화 변환, 해외 배송비 | 높음 |
| 지역별 커스터마이징 | 지역별 정비소, 판매자, 규제 대응 | 높음 |
| AWS 멀티리전 | 미국/유럽 리전 추가 | 높음 |

## 기술 스택 전환 계획

SCOOP의 현재 기술 스택에서 Loop 목표 스택으로의 전환은 점진적으로 진행한다. 한 번에 모든 것을 바꾸지 않고, Phase별로 필요한 부분만 전환하는 것이 핵심이다.

| 항목 | SCOOP 현재 | Loop 목표 | 전환 시점 |
|------|-----------|----------|----------|
| DB | MySQL (GCP) | PostgreSQL (AWS RDS) | Phase 1 시작 시 |
| 인프라 | GCP Cloud Run | AWS EC2 + ALB | Phase 1 시작 시 |
| 캐시 | 없음 | Redis (ElastiCache) | Phase 3 시작 시 |
| 실시간 | 없음 | WebSocket + Redis Pub/Sub | Phase 3 시작 시 |
| 결제 | 없음 | Stripe + Stripe Connect | Phase 1 시작 시 |
| CDN | 없음 | CloudFront | Phase 1 시작 시 |
| 프론트엔드 | React Native (Expo) | React Native (Expo) 유지 | 변경 없음 |
| API | tRPC | RESTful + tRPC 병행 | 점진적 전환 |
